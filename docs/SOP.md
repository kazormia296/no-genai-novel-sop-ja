# 1. 目的

本手順書は、調査、資料選定、着想、構成、執筆、推敲、校正、組版引渡しおよび記録保存において、生成AIまたはAIによる情報媒介の利用を、あらかじめ定めた保証範囲に従って排除・管理するための正式な作業手順を定める。

本手順書の目的は、単に作者が対話型AIを起動しなかったことを示すだけではない。次の事項を、対象範囲に応じて検証可能な形で記録することを目的とする。

1. 生成AI出力が本文、構成、設定、調査ノートまたは推敲に混入していないこと。
2. 調査資料および着想の直接的な源が記録されていること。
3. 制作中に学習済みモデルまたは生成機能が実行されていないこと。
4. 協力者による生成AI利用が管理されていること。
5. 制作環境、資料、草稿および最終成果物の完全性が記録されていること。
6. 保証できない範囲を明示し、過大な表示を防止すること。

# 2. 適用範囲

## 2.1 対象工程

本手順書は、原則として次の工程に適用する。

- 作品方針の確定
- 資料探索および資料入手
- 調査および事実確認
- 着想記録
- 登場人物、舞台、世界設定および筋の設計
- 初稿執筆
- 改稿および推敲
- 人間による講評、校閲および校正
- 最終原稿の確定
- 出版社または組版担当者への引渡し
- 制作記録の保存

## 2.2 対象外にできる工程

次の工程は、プロジェクト宣言書で明示した場合に限り、対象外とすることができる。

- 最終原稿確定後の機械的なファイル変換
- 出版社による組版、広告、書誌情報作成または販売促進
- 印刷所、電子書籍配信事業者または流通事業者の内部工程
- CPU、ファームウェア、ストレージ制御装置、製造設備、電力・通信インフラ
- OS、エディタ、IME、辞書またはライブラリの開発時における生成AI利用

対象外とした工程について、AI非使用を保証する表現を用いてはならない。

## 2.3 適用単位

適用単位は、作品、シリーズ、巻、短編または制作期間のいずれかとする。適用単位は `POLICY/PROJECT_DECLARATION.md` に明記しなければならない。

- **任意**：実施してもしなくてもよいが、実施した場合は記録する。

# 4. 用語および定義

## 4.1 生成AI

入力に応じて文章、画像、音声、映像、コード、要約、翻訳、構成案、候補文、校正案その他の新たな内容を生成する機械学習システムをいう。

## 4.2 AI媒介

AIまたは機械学習が、情報の推薦、順位付け、要約、分類、候補提示、補完、翻訳、検索結果の選別または評価を行い、作者が接する情報または選択肢を左右することをいう。

## 4.3 学習済みモデル

機械学習により作成された重み、パラメータ、統計モデル、報酬モデル、埋め込みモデル、言語モデル、認識モデルまたはこれらに準ずるデータをいう。

## 4.4 承認資料

資料責任者が来歴を確認し、資料台帳へ登録し、対象作品での利用を承認した資料をいう。

## 4.5 制作端末

調査ノート、着想記録、構成、本文、推敲記録または最終原稿を作成・保存する専用端末をいう。

## 4.6 外部端末

インターネット接続、資料入手、出版社との通信その他の外部作業に使用する端末をいう。制作端末と同一にしてはならない。

## 4.7 来歴

資料、アイデア、文章、ソフトウェア、辞書、設定またはファイルが、誰によって、いつ、何を基に、どの工程を経て作成・変更されたかを示す記録をいう。

## 4.8 不明資料

人間作成、AI生成、AI加工または選定経路を合理的に確認できない資料をいう。

## 4.9 例外

本手順書またはプロジェクト宣言書に定める条件から逸脱した事象をいう。意図的か偶発的かを問わない。

## 4.10 歴史的カットオフ

指定したUTC日時を境界として、それより後に公開、更新、生成またはビルドされた構成物を保証対象環境へ受け入れない統制をいう。カットオフは、締切以前のAI利用、決定論的コード生成、自動生成ファイルまたは不明な人間作業まで不存在であることを意味しない。

# 5. 保証プロファイル

プロジェクト開始前に、次のいずれかの保証プロファイルを選択しなければならない。独自条件を採用する場合は、各保証軸について対象範囲を明記する。

## 5.1 P1：主体的非使用プロファイル

作者および協力者が、対象作品について生成AIを意図的に起動せず、生成AI出力を本文等へ採用しない。

保証対象：

- 作者および協力者による明示的な生成AI利用
- 既知の生成AI出力の本文等への採用

保証対象外の例：

- 検索順位、推薦、IME候補の内部実装
- OS、ソフトウェアまたは辞書の開発時の生成AI利用
- 作者が過去に接したAI由来情報の無意識的影響

## 5.2 P2：閉鎖制作プロファイル

承認済み資料群、オフライン制作端末、固定辞書および人間協力者だけで制作する。本手順書の標準プロファイルとする。

必須条件：

- 制作端末は物理的にネットワークから隔離する。
- 制作端末では学習済みモデルを実行しない。
- 調査資料は資料台帳に登録する。
- 着想は着想台帳に記録する。
- 生成的校正、機械翻訳、AI要約、OCR、音声文字起こしを使用しない。
- 協力者は作品単位のAI非使用確認書を提出する。

## 5.3 P3：AI媒介排除プロファイル

P2に加え、資料発見、資料選定、候補提示および情報の順位付けにもAIを介在させない。

追加必須条件：

- 一般検索の推薦結果、AI回答欄、要約欄および自動候補を使用しない。
- 資料は、紙の参考文献、図書館分類、著者名、書名、ISBN、文書番号、固定URLまたは人間司書の案内から取得する。
- 検索システムを使う場合は、完全一致、項目指定または日付順等の非推薦的な照合方法に限定する。
- 人間協力者が資料やアイデアを提示する場合、その提示について生成AIを使用していないことを確認する。

## 5.4 P4：ソフトウェア供給網監査プロファイル

P2またはP3に加え、制作端末で使用するソフトウェア、辞書、依存部品およびビルド工程を監査する。

追加必須条件：

- 正確なソース版、依存関係、ハッシュおよびビルド手順を記録する。
- 可能な部品について再現可能ビルドを行う。
- ソースコードおよび辞書データの生成AI利用について作成者の表明を取得する。
- 表明または来歴を確認できない推移的依存部品は、不適合または保証対象外として明記する。

P4は、単にオープンソースであることをもって適合とはしない。

## 5.5 P4-H：歴史的カットオフ環境プロファイル

P4を基礎とし、制作環境の構成物を、指定した歴史的カットオフ以前に公開・保存された証拠付き構成物へ限定する。P4で求める作成者の生成AI利用表明は、構成物が明示した排除対象技術の合理的な利用可能時点より前に確定していたことを独立証拠で示せる範囲に限り、歴史的カットオフ証拠で代替できる。

追加必須条件：

- カットオフ日時を秒までのUTC日時で固定し、排除対象とする生成AIまたはコード生成技術、および日時選定根拠を記録する。
- OSの名称やリリース日だけで判定せず、カーネル、ユーザー空間、インストーラー、全パッケージ、依存部品、ツールチェーン、辞書、フォント、設定およびビルド種を個別に記録する。
- 各構成物について、締切以前の公開または独立保存を示す記録、取得元、署名・証明情報およびSHA-256を保存する。
- 排除対象技術ごとに合理的な利用可能時点と根拠資料を記録し、カットオフがそれより前であることを確認する。
- 現在のツールで古いソースを再構築した成果物を、無条件にP4-H適合としてはならない。再構築する場合は、ビルド環境とブートストラップ種もカットオフ条件へ含める。
- 制作開始後は、締切後の更新、実行物、辞書、フォント、設定または未登録媒体を導入しない。
- ファームウェア、マイクロコードおよびハードウェアを保証対象へ含めるか、対象外とするかを明記する。
- 締切以前のAI利用まで不存在であるとの無限定な表示を行わない。

P4-Hは、締切日以後の混入経路を因果的・手続的に閉じるための保証であり、「ソースの全行が人間だけで書かれた」と単独で証明するものではない。

## 5.6 P5：非電子創作プロファイル

調査、着想、構成、執筆および推敲を紙、筆記具、物理カード、紙辞書、印刷資料等のみで行う。デジタル化は、最終原稿確定後の複製工程として分離する。

P5は、制作ソフトウェアの供給網を創作工程から外すための最も厳格な運用である。ただし、印刷物の作成、紙資料の制作、最終転記等における間接的なAI利用まで無条件に保証するものではない。

# 6. 役割および責任

一人で制作する場合は、同一人物が複数の役割を兼務してよい。ただし、承認・監査の独立性を主張する場合は兼務してはならない。

| 役割 | 主な責任 |
|---|---|
| 制作責任者 | 適用範囲、保証プロファイル、開始日、例外判断および最終表示の承認 |
| 作者 | 調査、着想、構成、執筆、推敲および作業記録の作成 |
| 環境管理者 | 制作端末の構築、隔離、ソフトウェア凍結、ハッシュ記録および復旧試験 |
| 資料責任者 | 資料の来歴確認、分類、登録、追加承認および不明資料の隔離 |
| 校正責任者 | 校正者のAI非使用確認、指摘の受入れおよび反映記録 |
| 監査者 | 記録、例外、最終原稿および表示文の整合性確認 |

# 7. 標準システム構成

本章はP2の標準構成を定める。P1では一部を省略できる。P3、P4およびP4-Hでは追加統制を適用する。

## 7.1 機器

### 7.1.1 制作端末

制作端末は、次の条件を満たさなければならない。

- 対象作品専用または専用ユーザー領域であること。
- Wi-Fi、Bluetooth、携帯通信および有線ネットワークを物理的に使用不能とすること。
- 無線モジュールを取り外せる場合は取り外すこと。
- Ethernetケーブル、ドッキングステーションその他の通信手段を制作場所に常設しないこと。
- クラウド同期、ブラウザ、メール、チャットおよび遠隔操作機能を使用しないこと。
- USB直結の有線キーボードおよび必要に応じて有線マウスを使用すること。
- ネットワークプリンターを使用しないこと。

### 7.1.2 外部端末

外部端末は、資料入手、通信およびファイル受領に使用する。本文、構成、着想または推敲を外部端末上で作成してはならない。

### 7.1.3 周辺機器

標準構成は次のとおりとする。

- USB直結プリンター
- 入力専用の移送媒体
- 出力専用の移送媒体
- バックアップ用外部記憶媒体2本以上
- 一回書き込み光学媒体または封印可能な保管媒体
- 紙の国語辞典、漢和辞典、類語辞典および必要分野の専門辞典
- 紙の情報カード、原稿用紙、校正用紙および保管箱

## 7.2 推奨ソフトウェア

次のソフトウェアまたは同等の機能を持つ監査済みソフトウェアを使用する。

| 用途 | 推奨例 | 使用条件 |
|---|---|---|
| OS | Debian安定版の最小構成 | 完全インストール媒体からオフライン導入し、制作開始時に版を凍結する |
| エディタ | GNU Emacs | `-Q` 等で未監査設定を読み込まず、承認済み設定だけを使用する |
| 日本語入力 | DDSKKと固定ローカル辞書 | 辞書サーバ、Web検索、クラウド候補、予測・適応拡張を使用しない |
| 履歴管理 | Git | ローカルリポジトリのみ。リモートを登録しない |
| 差分確認 | `git diff`、GNU diffutils | 差分表示だけに使用する |
| 固定規則検査 | `grep`、`sed`、`awk`、`sort`、`uniq`、`wc` | あらかじめ人間が定めた文字列または規則だけを処理する |
| 完全性確認 | `sha256sum` | ファイル変更検知に使用する |
| 印刷 | ローカルUSB印刷 | クラウド印刷、ネットワーク印刷を禁止する |

特定の製品名を採用すること自体は、ソースコードまたは辞書が生成AI不使用であることを保証しない。P4またはP4-Hを選択しない限り、その点は保証対象外として記録する。

## 7.3 禁止ソフトウェアおよび禁止機能

制作端末では、次を導入または使用してはならない。

- 対話型AI、文章生成、画像生成またはコード生成サービス
- LLM、埋め込みモデル、報酬モデルまたは画像・音声認識モデルの実行環境
- AI文章校正、言い換え、文体改善、続きを書く機能
- 機械翻訳
- AI要約
- OCR
- 音声文字起こし
- 次文予測、スマート補完または生成的自動補完
- AI検索回答、検索結果要約、推薦欄または自動生成スニペット
- クラウドIME、クラウド辞書またはオンライン辞書
- クラウドストレージ、オンライン文書編集または同期サービス
- ブラウザ、メールクライアント、チャットクライアントまたはRSS取得機能
- 未承認のエディタ拡張、マクロ、プラグインまたは外部スクリプト
- 制作期間中の自動更新

## 7.4 日本語入力

日本語入力は、次の優先順で選択する。

1. **固定ローカルSKK辞書**：P2の標準。
2. **人間が編纂したプロジェクト専用小規模辞書**：P3または厳格な着想管理向け。
3. **IMEを使用しないUnicode直接入力**：辞書供給網まで避ける場合。
4. **手書き**：P5。

固定ローカル辞書を使用する場合は、辞書ファイル名、版、入手元、入手日、SHA-256、編纂主体、AI由来データの確認状況を `MANIFEST/DICTIONARY.tsv` に記録する。

個人辞書は対象作品専用とし、変更履歴を本文と同様に管理する。

# 9. 作業手順

## 9.1 工程0：プロジェクト宣言

制作責任者は、作業開始前に `POLICY/PROJECT_DECLARATION.md` を作成し、次を明記する。

- 作品名およびプロジェクトID
- 適用単位
- 選択した保証プロファイル
- AI非使用期間の開始日時
- 対象工程および対象外工程
- 禁止するAI機能およびAI媒介
- 検索順位、IME、OS、ソースコード、ハードウェアの扱い
- 既存の着想、メモまたは資料を持ち込む条件
- 不明資料の扱い
- 協力者の範囲
- 例外承認者
- 公表時に使用できる表示文の上限

開始日前に生成AIを用いて作成したメモ、構成、文章、検索結果、要約またはアイデアは、原則として対象作品へ持ち込んではならない。持ち込む場合は、対象作品を「生成AI不使用」と表示してはならず、例外記録へ明記する。

## 9.2 工程1：制作環境の構築および凍結

環境管理者は、次の順序で制作端末を構築する。

1. 制作端末を初期化する。
2. 外部端末で、承認済みOSおよび必要ソフトウェアの完全インストール媒体を準備する。
3. 公開された署名またはチェックサムを人間が確認する。
4. 制作端末をネットワークへ接続せずにOSを導入する。
5. 必要最小限のソフトウェアだけを導入する。
6. ブラウザ、メール、クラウド同期、AI機能および未承認サービスを削除または無効化する。
7. 無線モジュールを取り外し、有線ネットワーク接続手段を制作場所から排除する。
8. `MANIFEST/HARDWARE.tsv` および `MANIFEST/SOFTWARE.tsv` を作成する。
9. 設定ファイル、辞書および主要実行ファイルのハッシュを記録する。
10. 保存、印刷、差分確認、バックアップおよび復元を試験する。
11. 試験完了後、環境を凍結する。
12. 制作期間中のソフトウェア追加、更新または設定変更を禁止する。

やむを得ず更新する場合は、いったん制作を停止し、工程1を再実施して新しい環境版を付与する。

### 9.2.1 Emacs運用

Emacsを使用する場合は、未監査の個人設定、サイト設定および外部パッケージを読み込まない起動方法を用いる。例：

```sh
emacs -Q --no-desktop --load /absolute/path/to/approved-init.el
```

承認済み設定に含めてよい機能は、次に限定する。

- UTF-8プレーンテキストの読み書き
- ローカル日本語入力
- 自動保存およびバックアップ
- 行番号表示
- 画面上の折返し
- 文字数集計
- ローカルGit連携

次の機能は含めてはならない。

- ネットワーク接続
- 外部パッケージ取得
- LLM連携
- Web検索
- オンライン辞書
- 自動翻訳
- 文章生成または文章評価
- 未承認の外部プロセス呼出し

### 9.2.2 Git運用

Gitはローカル履歴管理だけに用いる。

```sh
git init
```

必須条件：

- リモートを登録しない。
- Gitホスティングサービスへ送信しない。
- Gitフックを使用しない。
- 一執筆日につき最低1回コミットする。
- 大規模改稿前に明示的な節目を記録する。
- コミットメッセージは作業種別と対象章を示す。

例：

```text
初稿：第03章
改稿：第03章 時系列整合
校正反映：校正者P02 指摘票12-18
```

## 9.3 工程2：資料探索および資料受入れ

### 9.3.1 資料探索

P2では、一般検索を用いる場合でも、AI回答、AI要約、推薦文および生成スニペットを読んではならない。検索結果から得た説明文を調査ノートへ転記してはならず、原資料を直接確認する。

P3では、次の方法に限定する。

- 紙の参考文献一覧
- 図書館の分類棚
- 著者名、書名、ISBNまたは公文書番号による完全一致検索
- 固定URLまたは既知の公式資料一覧
- 人間司書またはAI非使用を確認した専門家の案内
- 年代順、書名順または著者順による機械的な並び替え

### 9.3.2 資料分類

資料責任者は、各資料を次のいずれかに分類する。

| 区分 | 定義 | P1 | P2 | P3以上 |
|---|---|---:|---:|---:|
| H | 人間作成を合理的に確認できる | 使用可 | 使用可 | 使用可 |
| U | AI利用または作成主体が不明 | 開示の上で使用可 | 原則不可 | 使用禁止 |
| A | AI生成またはAI加工を確認 | 使用禁止 | 使用禁止 | 使用禁止 |

「人間らしい文体」だけをH判定の根拠としてはならない。刊行時期、版、著者、発行主体、原本、旧版、公式説明または著者確認等を組み合わせて判断する。

### 9.3.3 資料台帳

承認資料は `CORPUS/SOURCES.tsv` に登録する。必須項目は次のとおりとする。

```text
SOURCE_ID
AUTHOR
TITLE
PUBLISHER_OR_ISSUER
PUBLICATION_DATE
EDITION
ACQUISITION_PATH
DISCOVERY_METHOD
MEDIUM
PAGES_OR_SECTION
HUMAN_AUTHORSHIP_STATUS
GENAI_STATUS
FILE_SHA256
PERMITTED_USE
REVIEWER
REVIEW_DATE
NOTES
```

### 9.3.4 制作端末への移送

資料の移送は、次の優先順とする。

1. 紙資料のまま使用する。
2. 一回書き込み媒体で、承認済みファイルだけを移送する。
3. 入力専用媒体を使用し、制作端末からその媒体へ書き戻さない。

移送するファイル形式は、プレーンテキスト、画像または事前に安全確認した固定形式に限定する。OCR済みテキスト、AI要約、機械翻訳文または自動文字起こしを移送してはならない。

## 9.4 工程3：調査ノート作成

調査ノートは一行または一段落ごとに、次の区分を付す。

```text
[QUOTE]     原文の短い引用
[FACT]      資料が述べる事実
[INFERENCE] 作者自身の推論
[QUESTION]  未解決事項
[IDEA]      作品への応用候補
```

各記録には、資料IDおよびページ、節、時刻または位置情報を付す。

例：

```text
[FACT][S0012 p.84] 当時の郵便配達は冬季に経路を変更した。
[INFERENCE][S0012 p.84] 山間部では配達間隔が長くなった可能性がある。
[QUESTION] 冬季の具体的な配達頻度を別資料で確認する。
```

調査ノートに、出典のない外部記憶、検索結果の説明文または未登録資料の内容を混在させてはならない。

外国語資料を使用する場合は、人間が読解するか、AI非使用を確認した人間翻訳者へ依頼する。機械翻訳を使用してはならない。

## 9.5 工程4：着想の来歴管理

著作権法上保護されない可能性がある抽象的アイデアであっても、本手順書では来歴管理の対象とする。

各アイデアには `I0001` 等のIDを付し、`IDEAS/IDEAS.tsv` および個別カードへ次を記録する。

```text
IDEA_ID
DATE
PLACE
DESCRIPTION
DIRECT_TRIGGER
SOURCE_ID_OR_PERSON_ID
SOURCE_CATEGORY
AUTHOR_MODIFICATION
GENAI_RISK
STATUS
USED_IN
REVIEWER
NOTES
```

着想源は次のように分類する。

| 符号 | 着想源 | P2 | P3以上 |
|---|---|---:|---:|
| O | 直接観察 | 使用可 | 使用可 |
| E | 自分の経験 | 使用可 | 使用可 |
| D | 自分の日記または夢 | 使用可 | 使用可 |
| R | 承認資料 | 使用可 | 使用可 |
| H | AI非使用を確認した人間との会話 | 使用可 | 使用可 |
| C | 人間作成カードと物理サイコロ等による偶然操作 | 使用可 | 使用可 |
| M | 出所を特定できない記憶 | 開示の上で使用可 | 原則使用禁止 |
| A | AI出力またはAI由来と判明した着想 | 使用禁止 | 使用禁止 |

他者との会話から着想を得る場合、その人物が当該提案について生成AIを使用していないことを確認する。

物理的な偶然操作を用いる場合、語句カード、表、選択肢またはルール自体も人間作成でなければならない。

## 9.6 工程5：構成作成

構成は、紙カードまたは制作端末上のプレーンテキストで作成する。

各場面には次を付す。

- 場面ID
- 視点人物
- 時刻および場所
- 使用する着想ID
- 参照する資料ID
- 場面の目的
- 前後関係
- 未確認事項

構成ソフトの自動提案、プロット生成、類似作品推薦、感情曲線分析または人物関係の自動補完を使用してはならない。

## 9.7 工程6：執筆セッション

作者は、各執筆セッションの開始時に `REVIEW/SESSION_LOG.tsv` へ次を記録する。

```text
SESSION_ID
START_TIME
END_TIME
CHAPTER_OR_SCENE
SOURCE_IDS
IDEA_IDS
DEVICE_ID
SOFTWARE_BASELINE
DICTIONARY_VERSION
EXCEPTION_STATUS
NOTES
```

執筆中に新たな情報が必要になった場合、外部端末で即時検索してはならない。本文へ次のような保留記号を入れ、調査工程へ戻す。

```text
［要確認：冬季の船便頻度］
［固有名未確定］
［資料S0041を確認］
```

追加調査を行う場合は、執筆をいったん停止し、工程2および工程3を実施して資料群を再凍結する。

執筆終了時には、次を実施する。

1. ファイルを保存する。
2. 当日の差分を確認する。
3. セッションログを完了する。
4. Gitへコミットする。
5. 定めた頻度に従ってバックアップする。

## 9.8 工程7：機械的検査

制作端末で許可する検査は、人間が事前に定めた固定文字列または決定論的規則に限定する。

許可例：

- 禁止語句の完全一致検索
- 人名、地名および固有名詞の表記揺れ
- 連続する句読点
- 括弧の対応
- タブ、制御文字または不要な空白
- 章番号の連続性
- 文字数、行数および出現回数
- 前版との差分

禁止例：

- 自然な文章かどうかの評価
- 文体、感情、読みやすさまたは独創性の自動評価
- 言い換え、要約、続きの候補または改善案
- 形態素解析結果に基づく創作的な推奨
- 類似作品または既存作品との自動比較

実施した規則、規則ファイルの版および結果を記録する。

## 9.9 工程8：紙面推敲

推敲は、原則として印刷物上で実施する。

1. 対象原稿を印刷する。
2. 原稿の版、ハッシュ、印刷日および担当者を表紙へ記載する。
3. 削除、追加、移動および確認事項を色分けして記入する。
4. 大きな修正には理由を付す。
5. 制作端末へ人間が反映する。
6. Git差分により反映範囲を確認する。
7. 紙原稿を `REVIEW/PRINTED_PROOFS/` に対応する番号で保管する。

紙面推敲中に新たな資料や他者の提案を用いた場合は、その来歴を追加登録する。

## 9.10 工程9：人間による講評、校閲および校正

協力者には、作業開始前に次の条件を示し、署名または記録可能な同意を得る。

> 本作品の講評、校閲または校正において、生成AI、AI検索回答、AI要約、機械翻訳、文章校正AI、音声文字起こしAIおよび画像生成AIを使用しない。原稿の全部または一部を外部サービスへ送信しない。指摘は紙または承認済みプレーンテキスト形式で返却し、使用した道具を申告する。

協力者ごとに次を記録する。

```text
COLLABORATOR_ID
NAME
ROLE
RECEIVED_FILE_SHA256
TOOLS_USED
GENAI_NON_USE_CONFIRMED
DATE
RETURN_FORMAT
SIGNATURE_OR_CONFIRMATION
NOTES
```

校正者が生成AIを使用したことが判明した場合、その指摘およびそこから派生した変更は不適合として隔離する。

## 9.11 工程10：最終原稿確定

最終成果物は、原則として次の三つに分ける。

1. `RELEASE/NOVEL_FINAL.txt` — 本文
2. `RELEASE/FORMATTING.txt` — ルビ、改ページ、圏点、図版位置等の指定
3. `RELEASE/DECLARATION.md` — AI非使用範囲および対象外範囲の表示

本文ファイルには、可能な限り内容以外の自動生成メタデータを含めない。

組版指定例：

```text
ルビ：｜漢字《かんじ》
改ページ：［改ページ］
圏点：［圏点開始］……［圏点終了］
縦中横：［縦中横：12］
図版位置：［図版01］
```

DOCX、PDF、EPUBその他の形式が必要な場合は、次のいずれかを選択する。

- 出版社側の工程として対象外にする。
- 変換専用端末で、内容を変更しない機械的な変換だけを行う。
- AI非使用を確認した人間の組版担当者へ依頼する。

変換中に表現を変更した場合、その変換工程は創作工程に含めなければならない。

## 9.12 工程11：最終監査および公開承認

監査者は、公開前に次を確認する。

- プロジェクト宣言書が署名済みである。
- 選択した保証プロファイルと実際の運用が一致する。
- ハードウェア、ソフトウェア、辞書および設定台帳が完成している。
- 制作端末のネットワーク隔離記録がある。
- 全資料が資料台帳に登録されている。
- 採用した主要アイデアが着想台帳に記録されている。
- 全協力者のAI非使用確認がある。
- 例外がすべて評価・処理済みである。
- 未確認事項および保留記号が残っていない。
- 最終原稿と宣言文のハッシュが記録されている。
- 公表文が保証範囲を超えていない。
- P4-Hを選択した場合、カットオフ台帳、構築方式、ブートストラップ種、ファームウェア境界および締切後入力の不存在が確認されている。

監査完了後、制作責任者および監査者が `RELEASE/DECLARATION.md` を承認する。

# 10. バックアップおよび完全性管理

## 10.1 保存原則

最低3系統で保存する。

| コピー | 保管先 |
|---|---|
| 1 | 制作端末内蔵ストレージ |
| 2 | 制作場所内の外部記憶媒体 |
| 3 | 別の場所に保管する外部媒体または一回書き込み媒体 |

入力専用媒体、出力専用媒体およびバックアップ媒体は兼用してはならない。

## 10.2 ハッシュ

節目ごとに主要ファイルのSHA-256を記録する。例：

```sh
tmp="$(mktemp)"
find . -type f ! -path './MANIFEST/SHA256SUMS' -print0 \
  | LC_ALL=C sort -z \
  | xargs -0 sha256sum > "$tmp"
mv "$tmp" MANIFEST/SHA256SUMS
```

コマンドは環境管理者が対象環境で事前検証すること。ハッシュは、人間による著作またはAI非使用を証明するものではない。記録後にファイルが変更されていないことを確認するために使用する。

## 10.3 復元試験

少なくとも次の時点で、別媒体からの復元試験を行う。

- 制作開始前
- 大規模改稿完了後
- 最終原稿確定前
- 引渡し前

復元したファイルのハッシュが記録と一致することを確認する。

# 11. 例外および事故対応

すべての例外は `POLICY/EXCEPTIONS.tsv` に記録する。

必須項目：

```text
EXCEPTION_ID
DATE_TIME
DISCOVERED_BY
EVENT_TYPE
DESCRIPTION
AFFECTED_FILES
AFFECTED_SESSIONS
AFFECTED_SOURCES_OR_IDEAS
IMMEDIATE_ACTION
IMPACT_ASSESSMENT
CORRECTIVE_ACTION
DISPOSITION
APPROVER
CLOSURE_DATE
NOTES
```

## 11.1 AI出力を偶発的に閲覧した場合

1. 閲覧を直ちに中止する。
2. 表示内容を作品資料へ転記しない。
3. 閲覧時刻、内容の種類および影響を受け得る作業を記録する。
4. 直後に作成したメモ、構成または文章を隔離する。
5. 当該出力の影響を排除できるか評価する。
6. 排除できない場合は、影響を受けた部分を破棄して、承認資料だけから再作成する。
7. 厳格プロファイルでは、影響を受けたセッション以降の着想を再評価する。

## 11.2 制作端末がネットワークへ接続された場合

1. 直ちに物理切断する。
2. 制作を停止する。
3. 接続日時、接続先、通信の有無および原因を記録する。
4. ファイルおよび設定の現状ハッシュを取得する。
5. 環境管理者が不正変更、同期、更新または外部送信の可能性を評価する。
6. 必要に応じて、承認済み媒体から制作端末を再構築する。
7. 再構築後に新しい環境版を付与する。

## 11.3 不明資料が混入した場合

1. 資料をUに再分類する。
2. その資料から作成したノート、アイデアおよび文章を特定する。
3. 影響物を隔離する。
4. H資料から独立に再確認できる内容だけを再作成する。
5. 再作成できない部分を削除する。

## 11.4 協力者がAIを使用した場合

1. 当該協力者の返却物を不適合として隔離する。
2. 返却物を基に行った修正を特定する。
3. 修正を差し戻すか、人間だけで再評価する。
4. 作品の非使用表示に与える影響を評価する。
5. 必要に応じて協力者を変更する。

## 11.5 未承認ソフトウェアまたは設定変更

1. 制作を停止する。
2. 変更前後の版、設定および影響範囲を記録する。
3. 承認済み基準へ復元するか、新環境として再認証する。
4. 変更期間中の作業成果を再評価する。

# 12. P4：ソフトウェア供給網監査の追加手順

P4を選択した場合、環境管理者は各ソフトウェア、辞書、フォント、ライブラリおよびビルドツールについて、次を記録する。

## 12.1 必須統制

- バイナリと対応ソースを記録する。
- 依存部品を推移的に列挙する。
- ビルド時に外部取得が発生しないよう固定する。
- 可能な限り独立した二環境でビルドし、成果物を比較する。

## 12.2 不明部品

作成者の表明、完全な依存関係またはデータ来歴を得られない部品は、次のいずれかとする。

1. 使用しない。
2. 同等の監査可能な部品へ置き換える。
3. 保証対象外として明示し、P4完全適合を主張しない。

再現可能ビルドは、指定したソースと成果物の対応を確認する手段であり、ソースが人間だけで書かれたことを単独で証明するものではない。

## 12.3 学習済みモデル

P4では、学習済みモデルを制作端末へ導入しないことを原則とする。モデルを導入する例外を認める場合は、少なくとも次を監査しなければならない。

- 事前学習データ
- 教師あり追加学習データ
- 人間選好データ
- 報酬モデル
- RLHFまたはRLAIF
- 合成データ
- 蒸留元モデル
- 評価データおよびモデル選抜基準
- オンライン学習または個人化データ

これらを完全に確認できないモデルは使用禁止とする。

# 13. P4-H：歴史的カットオフ環境の追加手順

P4-Hを選択した場合は、P4の構成物列挙、完全性、依存関係およびビルド対応の要件に加えて本章を適用する。作成者の生成AI利用表明を歴史的証拠で代替できるのは、本章13.3の条件を満たす排除対象技術だけである。P4-Hの対象は、プロジェクト宣言書で列挙した制作環境の構成物であり、社会全体、作者の記憶または締切以前の開発工程に関する無限定な非AI証明ではない。

## 13.1 カットオフ宣言

制作責任者は、環境構築前に次を記録し、承認しなければならない。

```text
CUTOFF_ID
CUTOFF_UTC
CUTOFF_BASIS
PROHIBITED_TECHNOLOGY_SCOPE
TECHNOLOGY_AVAILABLE_AT
CUTOFF_EVIDENCE_IDS
INCLUDED_COMPONENT_CLASSES
EXCLUDED_COMPONENT_CLASSES
BUILD_METHOD=H-BINARY|H-SOURCE
FIRMWARE_BOUNDARY=INCLUDED|EXCLUDED|PARTIAL
APPROVED_BY
APPROVED_AT
```

`CUTOFF_UTC` は `YYYY-MM-DDThh:mm:ssZ` 形式とする。「生成AI登場以前」「古いLinux」「ChatGPT以前」等の相対的または解釈が分かれる表現だけで済ませてはならない。

`CUTOFF_BASIS` には、何の混入を排除するためにその日時を選んだかを記録する。対象は、例えば対話型コード生成、LLM補完、生成AIによる翻訳または合成データ利用のように具体化する。対象技術ごとに、公開、提供、研究利用または限定利用を含む合理的な利用可能時点を `TECHNOLOGY_AVAILABLE_AT` に記録し、カットオフがそれより前であることを証拠IDとともに示す。利用可能時点が `UNKNOWN` の対象技術については、歴史的証拠を作成者表明の代替にしてはならない。非公開の先行利用まで絶対に不存在であったとは推定せず、根拠資料と保証上の限界を併記する。

## 13.2 対象境界

次の構成物クラスを個別に対象内または対象外として判定する。

| 構成物クラス | 例 |
|---|---|
| OS基盤 | カーネル、ユーザー空間、インストーラー、ブートローダー、標準ライブラリ |
| アプリケーション | エディタ、IME、Git、差分・検査ツール、印刷ツール |
| ビルド環境 | コンパイラ、アセンブラ、リンカ、ビルドシステム、ビルダーOS、コンテナまたはVM |
| データ | 辞書、フォント、ロケール、テンプレート、規則ファイル |
| ブートストラップ | 初期コンパイラ、生成済みソース、バイナリ種、ファームウェア・ブロブ |
| 装置内コード | UEFI、CPUマイクロコード、ストレージ、キーボード、プリンター等のファームウェア |
| ハードウェア | CPU、主記憶、基板、周辺機器および製造工程 |

OSディストリビューションの名称またはリリース日だけを、全構成物の適格性証拠としてはならない。各パッケージ、同梱データ、生成済みファイルおよびビルド入力を台帳へ登録する。

対象外としたクラスは、不適合ではなく保証境界として扱える。ただし、公表文で当該クラスの非混入を主張してはならない。

## 13.3 構成物の適格性

構成物は、次の全条件を満たす場合に限り適格とする。

1. 締切以前に公開されたこと、または独立したアーカイブへ保存されたことを示す日時付き証拠がある。
2. 使用するファイルのSHA-256が記録されている。
3. 署名、配布元メタデータ、複数の独立保存記録その他の方法で、現在取得したファイルと当時の構成物の同一性を合理的に確認できる。
4. バイナリを使用する場合は、対応するソース、版およびビルド来歴が記録されている。確認できない場合は、その不足を保証境界として明示し、ソースコードに関するP4-H完全適合を主張しない。
5. 締切後のパッチ、バックポート、再パッケージ、辞書更新または設定生成物が混在していない。

P4の作成者表明を歴史的証拠で代替する場合は、さらに次の全条件を満たす。

1. 排除対象技術が具体的に定義されている。
2. 対象技術の合理的な利用可能時点が根拠資料で示されている。
3. 構成物のソーススナップショットまたはバイナリの内容が、その利用可能時点より前に固定されていたことを独立証拠で示せる。
4. 代替する表明の範囲が当該技術だけに限定され、締切以前の別種のAI、自動生成または非公開先行利用まで不存在とは表示しない。

現在ダウンロードした古いISOまたはパッケージは、名称や内部タイムスタンプだけでは適格としない。当時の署名・ハッシュまたは独立した保存記録と照合する。製造・封印日時を合理的に確認できる書換不能な原時点媒体は、有力な保存証拠として扱える。

日時・版を指定して当時のDebianソースおよびバイナリを参照する場合は、[snapshot.debian.org](https://snapshot.debian.org/) のような公式履歴アーカイブを使用できる。使用したスナップショット日時とファイルハッシュを台帳へ固定する。

## 13.4 構築方式

### 13.4.1 H-BINARY：原時点バイナリ方式

締切以前に配布・保存された公式バイナリイメージを、そのまま導入する方式とする。

必須条件：

- イメージ、インストーラーおよび同梱パッケージの版、署名、ハッシュおよび保存証拠を記録する。
- インストール時およびインストール後にネットワーク取得を行わない。
- 締切後のセキュリティ更新、ドライバー、証明書、フォント、辞書または設定を追加しない。
- インストール完了後の基準ディスクイメージまたは全管理対象ファイルのハッシュを保存する。

H-BINARYは締切後のコード混入を強く排除できる一方、バイナリと全ソースの対応または初期ビルド環境を単独で証明しない。その不足をP4記録へ残し、対応ソースを確認できない範囲について「ソースコードがカットオフ適合」と表示しない。

### 13.4.2 H-SOURCE：歴史的ソース再構築方式

締切以前のソースから環境を再構築する方式とする。

必須条件：

- ソースだけでなく、ビルダーOS、コンパイラ、リンカ、ビルドシステム、依存バイナリ、生成済みソースおよびブートストラップ種も締切条件へ含める。
- ビルド中のネットワーク取得を禁止し、全入力を事前登録する。
- 独立して準備した二環境でビルドし、対象成果物のビット単位一致を確認する。
- 一致しない場合は差分と原因を解消するまでP4-H完全適合としない。
- 現代のコンパイラ、コンテナ、CI、パッケージマネージャまたは生成スクリプトを使った場合は、それを締切後入力として扱う。

再現可能ビルドの判定は、同じソース、ビルド環境および手順から同一成果物を得るという [Reproducible Buildsの定義](https://reproducible-builds.org/docs/formal-definition/) に従う。GNU/Linuxの構築が初期バイナリ種へ依存する問題については、[Bootstrappable Builds](https://www.bootstrappable.org/projects.html) の考え方に基づき、種の種類、版、ハッシュおよび対象外境界を記録する。

## 13.5 制作期間中の隔離

- 制作端末の無線モジュールを取り外し、有線通信経路を制作場所へ置かない。
- BIOSまたはUEFIでネットワークブートおよび不要な外部起動を無効化する。
- 自動更新、時刻同期、証明書更新、パッケージ取得および遠隔管理を実行しない。
- 搬入は事前登録した書換不能媒体を原則とし、搬入ファイルのハッシュと媒体IDを記録する。
- 搬入形式は原則としてUTF-8プレーンテキストとし、実行物、スクリプト、アーカイブ、PDF、Office文書、画像、動画、音声および新規フォントを持ち込まない。
- やむを得ず他形式を扱う場合は、使用パーサーと攻撃面を承認し、例外記録を作成する。
- 搬入媒体と搬出媒体を分離し、制作端末へ秘密情報または通常業務データを保存しない。

ファイアウォール、機内モードまたは接続しないという運用宣言だけを、物理的隔離の代替としてはならない。

## 13.6 ファームウェアおよびハードウェア

ファームウェアを対象内とする場合、OSと同じ締切、保存証拠、版およびハッシュ要件を適用する。対象外とする場合は、UEFI、CPUマイクロコード、ストレージ、入力装置、表示装置およびプリンターのどこからを対象外としたかを列挙する。

ハードウェアの製造日だけから、内蔵ファームウェアまたは製造工程のAI非使用を推定してはならない。ハードウェアを対象外としたP4-H適合は可能だが、表示文にその境界を含める。

## 13.7 歴史的カットオフ台帳

各構成物は `templates/HISTORICAL_CUTOFF_MANIFEST.tsv` に登録し、少なくとも次を記録する。

```text

CUTOFF_BASIS
PROHIBITED_TECHNOLOGY_SCOPE
TECHNOLOGY_AVAILABLE_AT
CUTOFF_EVIDENCE_IDS
```

## 13.8 セキュリティ上の制約

P4-Hは、意図的にセキュリティ更新を停止するため、ネットワーク接続、未信頼ファイルの解析または通常業務との兼用に適さない。専用・オフライン・最小権限・単純形式の環境としてのみ運用する。

古いOSの脆弱性をP4-Hの目的で正当化してはならない。隔離を維持できない場合はP4-Hを中止し、更新済み環境を用いるP2、P3またはP4へ保証プロファイルを変更する。

## 13.9 不適合および復旧

次のいずれかが発生した場合、当該環境をP4-H不適合として直ちに隔離する。

- 締切後の構成物、更新、ビルド入力または実行物を導入した。
- 公開日時、保存日時、ハッシュまたは取得元を確認できない構成物が見つかった。
- 制作端末がネットワークへ接続された。
- 未登録または書換可能な媒体からファイルを受け入れた。
- ファームウェア境界またはブートストラップ種が記録と一致しない。

復旧時は、承認済み基準イメージから再構築し、事故期間中の成果物を隔離して人間が影響を評価する。締切後入力を例外として保持したまま、無条件のP4-H完全適合を表示してはならない。必要に応じてP4へ格下げし、変更日と影響範囲を公表する。

# 14. P5：非電子創作の代替手順

P5では、工程0から工程10までを、次の非電子手段に置き換える。

## 14.1 使用物品

- 通し番号付き製本ノート
- 400字詰め原稿用紙または番号付き用紙
- 鉛筆、万年筆、消しゴムおよび色鉛筆
- 情報カードおよびカードボックス
- コルクボード

- 印刷済み承認資料
- カーボン紙または複写紙
- 物理的なサイコロまたはトランプ
- 鍵付き保管箱

## 14.2 作業順序

1. 紙のプロジェクト宣言書へ署名する。
2. 紙の資料台帳を作る。
3. 各資料へ資料IDを付す。
4. 調査ノートを手書きし、各記録へ資料IDとページを付す。
5. 着想カードを作る。
6. 構成カードを物理的に並べる。
7. 初稿を番号付き原稿用紙へ執筆する。
8. カーボン紙または複写により同時控えを作る。
9. 推敲および校正を紙だけで行う。
10. 最終紙原稿を確定し、全ページへ通し番号を付す。
11. 最終紙原稿確定後、必要に応じて独立したデジタル転記工程を行う。

## 14.3 デジタル転記

OCRを使用してはならない。

推奨手順：

1. 転記担当者Aが紙原稿を入力する。
2. 担当者Bが紙原稿と印刷出力を一文字ずつ照合する。
3. 相違を訂正記録へ記入する。
4. 内容変更を行わず、誤転記だけを修正する。
5. 完成ファイルのハッシュを記録する。

転記中に新たな表現変更を行った場合、デジタル転記は創作工程に含まれ、転記端末およびソフトウェアも保証対象となる。

# 15. 公表文の基準

「一切の生成AIを使用していない」という無限定の表現は、全工程、全資料、全協力者、全ソフトウェア供給網および間接的影響を合理的に保証できる場合を除き、使用してはならない。

## 15.1 P1向け表示例

> 本作の構想、調査、執筆および推敲において、作者は生成AIを意図的に起動せず、生成AIの出力を本文、構成、設定および調査メモに採用していない。検索順位、使用ソフトウェアの開発過程、OS、辞書およびハードウェアにおける間接的なAI利用は監査対象外である。

## 15.2 P2向け表示例

> 本作について、YYYY年MM月DD日から最終原稿確定までをAI非使用期間とした。期間中、制作端末を物理的にネットワークから切断し、学習済みモデル、生成機能、AI要約、AI推薦、機械翻訳、OCR、音声文字起こしおよび生成的校正を使用していない。調査は登録済み資料群に限定し、日本語入力には固定されたローカル辞書および作者自身の個人辞書を使用した。OS、エディタ、辞書およびハードウェアの開発時における間接的な生成AI利用は保証範囲外である。

## 15.3 P3向け表示例

> 本作の調査、着想、構成、執筆および推敲には、事前登録した人間作成資料、直接観察およびAI非使用を確認した人間協力者だけを用いた。資料の発見および選定には、紙の参考文献、図書館分類、完全一致書誌検索または人間による案内を用い、AI回答、AI要約、推薦および生成的候補提示を使用していない。制作端末では学習済みモデルを実行していない。

## 15.4 P4-H向け表示例

> 本作の制作環境には、歴史的カットオフ `YYYY-MM-DDThh:mm:ssZ` を適用し、排除対象技術を `<具体的な技術>` と定義した。OS、カーネル、ユーザー空間、制作ソフトウェア、辞書、フォント、ビルド環境および登録済み依存部品は、締切以前の公開・保存証拠とハッシュを確認した構成物へ固定し、制作開始後の更新を行っていない。構築方式は `H-BINARY／H-SOURCE` であり、制作端末は物理的にネットワークから隔離した。ファームウェアおよびハードウェアの保証境界は `<対象内／対象外の詳細>` である。本表示は、締切以前のAI利用、非公開先行利用または別種の自動生成まで不存在であることを保証しない。

## 15.5 P5向け表示例

> 本作の調査、着想、構成、表現の決定、執筆および推敲は、紙資料、筆記具、物理カードおよびAI非使用を確認した人間協力者だけで行った。最終紙原稿確定後のデジタル入力は、内容を変更しない複製工程として別に実施し、OCRおよび生成AIを使用していない。
